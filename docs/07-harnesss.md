# 07-harnesss.md — Harness 驾驭工程设计

## 1. 概念说明

### 1.1 什么是 Harness

Harness（驾驭工程）是一套 AI 协作开发的方法论和配套模板框架。在本项目中，Harness 定义了开发者与 AI 编码助手（Claude Code）之间的**协作协议**，确保：

- AI 的产出符合项目架构规范
- 代码变更始终在边界约束内
- 开发者对关键决策保持控制权
- 文档与代码同步演进

### 1.2 核心原则

| 原则 | 说明 |
|------|------|
| **文档驱动** | 架构决策先在文档中确认，再写代码 |
| **约束优先** | 边界和规范是硬性约束，不因便捷而妥协 |
| **渐进细化** | 从架构 → 设计 → 实现，逐步收敛 |
| **验证闭环** | 每步产出都有对应的验证环节 |

### 2.2 各节点 AI 参与度

| 阶段 | AI 角色 | 开发者角色 |
|------|---------|-----------|
| 架构设计 | 提出方案、权衡建议 | 决策确认 |
| 文档编写 | 撰写、格式化 | 审核、补充 |
| 代码实现 | 按文档生成代码 | 审查、修改 |

| 卫生清理 | 执行检查清单 | 确认清理结果 |

## 3. 各阶段 Prompt 模板

### 3.1 架构设计阶段

向 AI 描述需求时的模板：

```
## 需求描述
[简短描述要做什么]

## 约束条件
- 技术栈：Go 1.26.1 + gioui 0.10.0
- 遵循 docs/01-architecture.md 的分层架构
- 外部 API 调用必须通过 client/ 层
- 遵循 docs/03-boundary-constraints.md 的依赖方向

## 期望产出
- 设计方案（含包结构和数据流）
- 受影响的文件列表
- 测试策略
```

### 3.2 代码实现阶段

```
## 实现目标
[要实现的模块/功能]

## 参考文档
- docs/01-architecture.md §[相关章节]
- docs/02-coding-guidelines.md
- docs/03-boundary-constraints.md

## 约束
- 所有导出名称必须有文档注释
- 错误处理必须显式，不使用 _
- 使用 slog 日志
- 同时生成测试文件
```

### 3.3 代码审查阶段

```
## 审查范围
[文件列表或功能模块]

## 审查重点
- [ ] 是否符合 docs/02-coding-guidelines.md 编码规范
- [ ] 是否违反 docs/03-boundary-constraints.md 依赖规则
- [ ] 错误处理是否完整
- [ ] 测试覆盖率是否达标
- [ ] 是否存在代码异味
```

## 4. Skill 文件机制

### 4.1 Skill 定位

Skill 文件（位于 `docs/.skills/` 下）是可执行的开发任务模板。它定义了一个**标准化的操作流程**，开发者或 AI 助手可以按步骤执行。

### 4.2 已有 Skill

| Skill | 路径 | 触发条件 |
|-------|------|---------|
| 代码库卫生清理 | `docs/.skills/codebase-hygiene/SKILL.md` | 定期执行或发现代码异味时 |

### 4.3 编写新 Skill 的模板

```markdown
# SKILL.md — [技能名称]

## 元信息
- 名称: [技能标识]
- 用途: [一句话说明]
- 触发: [什么场景下执行此技能]

## 前置条件
- [需要满足的条件]

## 执行步骤
1. [步骤1标题]
   - [详细操作]
   - [验证方式]

2. [步骤2标题]
   - [详细操作]
   - [验证方式]

## 产出确认
- [ ] 检查项1
- [ ] 检查项2
```

## 5. 文档与代码同步机制

### 5.1 同步规则

| 触发条件 | 需要同步的文档 | 执行者 |
|---------|--------------|-------|
| 新增外部依赖 | `03-boundary-constraints.md` 外部包登记表 | 开发者 |
| 新增 API 调用 | `04-api-version-specification.md` 接口文档 | 开发者 |
| 新增错误码 | `05-errors-code.md` 错误码表 | 开发者 |
| 接口不兼容变更 | `04-api-version-specification.md` 版本记录 | 开发者 |

### 5.2 文档优先原则

修改代码前，**先更新文档**，再实现。流程如下：

```
1. 发现需要变更 → 2. 修改对应文档 → 3. 文档确认 → 4. 修改代码 → 5. 验证一致性
```

### 5.3 一致性验证

```bash
# 检查文档中声明的错误码是否在代码中实现
grep -r "var Err" internal/errors/ | sort > /tmp/code_errors.txt
grep -E "E[0-3][A-Z]{2}[0-9]{3}" docs/05-errors-code.md | sort > /tmp/doc_errors.txt
diff /tmp/code_errors.txt /tmp/doc_errors.txt
```

## 6. 健康检查

项目定期执行以下健康检查（建议每两周一次）：

- [ ] 所有文档是否与当前代码一致？
- [ ] 是否有违反包边界约束的代码？
- [ ] 测试覆盖率是否仍 ≥ 80%？
- [ ] 是否有未登记的 API 变更？
- [ ] 是否有未登记的错误码？
- [ ] CI 流水线是否正常运行？

执行健康检查时，运行 `docs/.skills/codebase-hygiene/SKILL.md` 中的流程。
